## What we are building
A natural language playlist generator powered by the user's own YouTube Music listening history.

Inspired by Spotify's AI DJ / natural language playlist feature — but for YouTube Music, which has no official equivalent.

**The core experience:**
User types a natural language query ("something for a Sunday morning that isn't too intense") → the tool retrieves songs from their personal YouTube Music history that match the vibe → LLM curates a final playlist with a short rationale per song.

**Why it's interesting as a portfolio piece:**
- Demonstrates RAG / retrieval over personal data (layer 3 of Anna's Applied AI portfolio)
- The "documents" are songs; the "knowledge base" is personal listening history; retrieval is semantic (by mood/activity/vibe); generation produces a curated playlist with explanations
- Personally meaningful and immediately demo-able
- Solves a real problem Anna has (YouTube Music has no AI playlist feature)

## Planned tech stack

### Data extraction
- **ytmusicapi** — unofficial Python library that reverse-engineers the YouTube Music web client
  - Can fetch: full listening history, liked songs, playlists
  - Works for personal use; actively maintained; could break if YouTube changes internal API (acceptable risk for a portfolio project)
  - Alternative: Google Takeout (simpler but less rich data)
  - Note: there is NO official YouTube Music API

### Enrichment
- For each song (title + artist): call **gpt-4o-mini** to generate a mood/vibe/activity description
  - e.g. "melancholic indie folk, good for late nights or long train journeys"
  - Alternative: **Last.fm API** (free) for genre tags if real metadata is preferred
  - Batch + cache to manage API costs (potentially hundreds of songs)

### Embeddings + vector storage
- **OpenAI embeddings** (text-embedding-ada-002 or text-embedding-3-small) on each song description
- Store in **Supabase with pgvector** extension
  - Same Supabase pattern used in AutoFlow — Anna is already familiar with this

### Query + generation
- User types natural language query
- Embed the query → vector similarity search → retrieve top N songs from history
- **gpt-4o-mini** curates final playlist (10-15 songs), writes one-line rationale per song

### Frontend
- **Lovable** — same tool used for AutoFlow
  - Text input for the query
  - Playlist output: song cards, each with title, artist, one-line reason it was included

## Architecture overview

```
Step 1 — EXTRACT
ytmusicapi → fetch listening history + liked songs
→ list of [title, artist, album, play_count, timestamps]

Step 2 — ENRICH
For each song: gpt-4o-mini generates mood/vibe description
→ "upbeat indie pop, good for morning commute or working out"
(batch processing, cache results)

Step 3 — EMBED + STORE
OpenAI embeddings on each description
→ Supabase + pgvector

Step 4 — QUERY (at runtime)
User types: "something for a Sunday morning that isn't too intense"
→ embed query → vector similarity search → top 20 songs retrieved

Step 5 — GENERATE (at runtime)
gpt-4o-mini: curate top 10-15, write rationale per song, explain playlist

Step 6 — RESULTS UI (Lovable)
Playlist cards: song + artist + one-line reason
Optional: playlist name generated by LLM
```

## Key decisions already made
- **Tool-agnostic where possible** — learned from AutoFlow that guessing specifics (tools, context) the user hasn't provided creates trust-breaking moments
- **gpt-4o-mini for both enrichment and generation** — cheap, fast, sufficient for this task
- **Supabase pgvector** — same stack as AutoFlow, no new infrastructure to learn
- **Lovable for frontend** — same tool as AutoFlow
